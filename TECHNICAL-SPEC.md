# cairn Documentation, Demos & Deployment — Technical Specification

## 1. Overview

This specification covers three workstreams that transform cairn from a library-only project into a polished, accessible ecosystem:

1. **Documentation Site** — A Docusaurus-powered site with versioned docs, language-switcher code snippets, and automated GitHub Pages deployment.
2. **Demo App Expansion & Dockerization** — Complete cross-language coverage for all demo apps, Docker images on `ghcr.io`, and frictionless `docker run` usage.
3. **Infrastructure Documentation** — Clear value propositions, deployment guides (including Cloudflare), and comprehensive server-node documentation.

### Primary Use Cases

- A developer discovers cairn and wants to understand what it does, install it, and see it work — all within 10 minutes.
- A developer wants to run a demo app without installing any language toolchain.
- An ops engineer wants to deploy signaling/relay infrastructure for their team.

### Target Platforms

- Documentation site: GitHub Pages (static)
- Demo Docker images: `linux/amd64` and `linux/arm64`
- Cloudflare deployment: Workers (signaling), Tunnel (relay)

### Non-Goals

- Rewriting existing library code or changing APIs
- Building a custom documentation framework
- Supporting Kubernetes or Helm charts (out of scope for this iteration)
- Writing new conformance tests

### 4.6 Documentation Content — Infrastructure

#### 4.6.1 `docs/infrastructure/overview.md`

**This is the critical "value proposition" page.** Structure:

Opening paragraph: "cairn works out of the box with zero infrastructure. Here's when and why you'd want to add your own servers."

**Tier comparison table**:

| | Tier 0 (Default) | Tier 1 (Signaling + Relay) | Tier 2 (Server Peer) |
|---|---|---|---|
| Setup | None | 2 Docker containers | 3 Docker containers |
| NAT traversal | Public STUN, best-effort | TURN relay, symmetric NAT | Full |
| Discovery speed | 5-30s (DHT/mDNS) | <1s (signaling) | <1s |
| Offline messages | No | No | Yes (store-and-forward) |
| Always-on relay | No | Yes | Yes |
| Multi-device sync | Manual | Manual | Automatic (hub) |
| Cost | Free | Free (Cloudflare) or ~$5/mo VPS | Same + storage |

**Decision flowchart** (text-based):
- "Are your peers on the same LAN?" → Tier 0 is fine
- "Do peers have public IPs or simple NAT?" → Tier 0 is fine
- "Are peers behind symmetric NAT or corporate firewalls?" → Tier 1
- "Do you need offline message delivery or multi-device sync?" → Tier 2

#### 4.6.5 `docs/infrastructure/cloudflare.md`

Cloudflare deployment guide:

**Signaling on Cloudflare Workers**:
- Cloudflare Workers support WebSockets natively
- Create a Worker that proxies WebSocket connections to a Durable Object or to a backend signaling server via Cloudflare Tunnel
- Step-by-step: create tunnel, configure DNS, run `cairn-signal` behind the tunnel
- Free tier compatibility (Workers free plan supports WebSockets)

**Relay via Cloudflare Tunnel**:
- TURN requires UDP, which Cloudflare Workers does not support
- Solution: run `cairn-relay` on a lightweight VM (e.g., free Oracle Cloud, $5 VPS) and expose it via Cloudflare Tunnel for the REST API, while the TURN UDP port is exposed directly
- Step-by-step: install `cloudflared`, create tunnel, configure DNS for the REST API, expose UDP port directly

**Server Node behind Cloudflare Tunnel**:
- Run `cairn-server` on a VPS, expose management API via Cloudflare Tunnel
- TLS handled by Cloudflare, no manual cert management needed

**Cost summary**: signaling is free (Cloudflare Workers free tier or tunnel to a $0 VM), relay needs a $0-5/mo VM with a public IP for UDP.

### 4.10 Demo App — Folder Sync Go Implementation

New file: `demo/folder-sync/go/main.go` (and supporting files).

Must implement the same CLI interface as the Rust and TypeScript versions:
- `--dir <path>` — directory to sync
- `--pair-pin` — initiator: display PIN
- `--enter-pin XXXX-XXXX` — responder: enter PIN
- `--pair-qr` / `--scan-qr` — QR pairing
- `--pair-link` / `--from-link <uri>` — link pairing
- `--mesh` — enable mesh routing
- `--server-hub` — always-on sync hub mode
- `--signal <url>` — signaling server URL
- `--turn <url>` — TURN relay URL
- `--verbose` — structured logging

**Sync protocol**: Match the existing Rust/TypeScript implementations:
- File watcher on the sync directory
- On change: compute SHA-256 hash, compare with peer's manifest
- Transfer changed files in 64KB chunks
- Delta sync: only send changed chunks
- Conflict resolution: rename conflicting file with `.conflict-{timestamp}` suffix

Use `github.com/moukrea/cairn/packages/go/cairn-p2p` as the P2P library.

### 4.13 Demo App Dockerfiles

Each demo language variant gets a Dockerfile optimized for minimal image size and fast startup.

**Dockerfile pattern for compiled languages (Rust, Go)**:
```dockerfile
# Builder stage
FROM {lang-image} AS builder
WORKDIR /build
COPY . .
RUN {build-command}

# Runtime stage
FROM debian:bookworm-slim
RUN apt-get update && apt-get install -y --no-install-recommends ca-certificates && rm -rf /var/lib/apt/lists/*
COPY --from=builder /build/{binary} /usr/local/bin/
ENTRYPOINT ["{binary}"]
```

**Dockerfile pattern for interpreted languages (TypeScript, Python, PHP)**:
```dockerfile
FROM {lang-image}
WORKDIR /app
COPY {dependency-file} .
RUN {install-command}
COPY . .
ENTRYPOINT ["{runtime}", "{entry-point}"]
```

**All demo Dockerfiles must**:
- Use multi-stage builds for compiled languages
- Include `ca-certificates` for TLS
- Set `ENTRYPOINT` to the demo binary/script
- Support `--signal` and `--turn` flags for optional server configuration
- Use `-it` compatible entrypoints (interactive terminal demos)

### 9.3 CI/CD — Documentation Workflow

`.github/workflows/docs.yml`:

```yaml
name: Deploy Documentation

on:
  push:
    branches: [main]
    paths:
      - 'website/**'

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
          cache-dependency-path: website/package-lock.json
      - run: cd website && npm ci
      - run: cd website && npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: website/build

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

## 10. Implementation Notes

### 10.1 Content Migration

The existing `docs/getting-started.md` content should be split across:
- `website/docs/getting-started/quick-start.md` — the step-by-step walkthrough (majority of content)
- `website/docs/getting-started/installation.md` — the prerequisites table

The existing `docs/design-doc.md`, `docs/PRD.md`, and `docs/technical-specification.md` are internal design documents and should NOT be migrated to the Docusaurus site. They remain in `docs/` as project references.

### 10.2 Code Examples

All code examples in documentation must be tested against the current API. Use the existing code from:
- `docs/getting-started.md` — verified quick-start examples
- `demo/messaging/*/` — verified demo code
- `README.md` — verified overview examples

Do not invent new API calls. If an API method is shown in documentation, it must match an existing method in the library.

### 10.3 Demo Implementation Approach

For new demo implementations (folder-sync in Go/Python/PHP):
- Read the existing Rust (`demo/folder-sync/rust/src/main.rs`, `sync.rs`) and TypeScript (`demo/folder-sync/typescript/src/index.ts`, `sync.ts`) implementations
- Port the logic faithfully — same sync protocol, same CLI flags, same conflict resolution
- Use the respective language's cairn-p2p library with the same API patterns shown in the messaging demo for that language

### 10.4 Docusaurus Version

Use Docusaurus v3.7+ (latest stable). Do not use Docusaurus v2.

### 10.5 README Update

After the documentation site is live, update the root `README.md`:
- Add a prominent link to the documentation site in the "Documentation" section
- Keep the README concise — it serves as a landing page, not comprehensive docs
- The existing quick-start code examples in the README should remain (they're good for GitHub visitors)

### 10.6 Security Considerations

- Demo Dockerfiles must not run as root in the final stage
- Demo Docker images must not embed secrets or credentials
- The Cloudflare deployment guide must emphasize: never expose management API tokens in client-side code

### 10.7 Known Complexity Areas

- **LanguageTabs component**: Must work correctly with Docusaurus MDX v3 parser. The `groupId` synchronization is a built-in Docusaurus feature but requires correct `<TabItem>` nesting.
- **Folder sync demo ports**: The sync protocol (chunked transfer, delta sync, conflict resolution) is non-trivial to port. Implementers should closely follow the existing Rust/TypeScript implementations.
- **Cloudflare Workers + WebSocket**: The signaling server can run behind a Cloudflare Tunnel rather than as a Worker, which is simpler and avoids Durable Objects complexity. The spec recommends the tunnel approach.
